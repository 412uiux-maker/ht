> **Как использовать:** `claude "Работай по docs/prompts/improvement.md — начни с P0"`

---

## 0. Контекст: что это и где мы

HappyTails — маркетплейс услуг и контента о питомцах для Узбекистана (uz/ru, оплата Click/Payme/Uzum). Монорепо (npm workspaces): `apps/{couples,vendor,admin,telegram-bot}`, `services/api` (Express + PostgreSQL), `packages/shared`.

Якорный поток (онлайн-консультация ветеринара) собран **end-to-end**: список врачей → запись → оплата (демо) → чат (polling 3s) → видеозвонок (WebRTC + WS) с входящим вызовом врачу и таймером консультации.

**Этот бриф не про новые фичи ради фич.** Он про то, чтобы довести уже построенное до уровня, на котором платформе можно доверять деньги и персональные данные: безопасность, реальные платежи, целостность модели заказа, консолидация общего кода и качество (тесты, a11y, языковой паритет).

- **Дизайн-система** (`DESIGN.md`): тёплая коралл/оранж палитра, токены, Onest, анти-референсы, motion-правила. Чёткая и последовательная.

- **Admin RBAC + статусная модель заказа** — согласованы со спекой (`created → paid → accepted → in_progress → completed → reviewed`, ветви refunded).
- **Брифы по приложениям** (`docs/prompts/*.md`) — хороший шаблон с Milestones и DoD; этот план переиспользует их.

Контракты API из брифов **менять нельзя** без явной причины — на них завязаны прототипы в `services/api/public/*.html`.

## 3. Находки аудита по серьёзности

### 🔴 P0 — Безопасность и доверие (блокеры для прод-данных)
1. **Пароли в открытом виде.** `vendor_credentials.password` и `admin_users.password` хранятся и сравниваются как plaintext (`WHERE password=$2`). Нет хеширования.
2. **Нет валидации Telegram `initData`.** Спека §7.4 требует HMAC-проверку подписи на сервере. Сейчас личность клиента = `owner_id` (случайный UUID из localStorage), который **можно подделать**.
3. **IDOR / нет проверки владения.** Эндпоинты `pets`, `consultations`, `orders` принимают `owner_id`/`id` из запроса без проверки, что ресурс принадлежит вызывающему. Любой может читать/менять чужие данные, зная id.
4. **Admin-auth слабый.** Логин/пароль шлются в заголовках `x-admin-email`/`x-admin-password` на каждый запрос (без токена/сессии/срока).
5. **Платёжный webhook без проверки подписи** (`/payments/webhook/:provider` — заглушка `received:true`).
6. **Утечки в ответах.** OTP-код возвращается в `_dev_code`, демо-пароли задокументированы и сидятся в БД.

### 🟠 P1 — Платежи и целостность заказа
7. **Нет реальных платежей.** Только `POST /payments/simulate`, создающий заказ сразу в статусе `paid`. Нет интеграций Click/Payme/Uzum (redirect/checkout), нет эскроу, нет реальных возвратов.
8. **Жизненный цикл заказа неполон.** Есть `GET /orders` и создание через simulate; **нет** переходов `accepted/in_progress/completed/cancelled/refunded` как единого источника истины. Статусы дублируются между `orders` и `consultations` без синхронизации.
9. **Модель данных разъехалась со спекой.** Нет сущностей User/Service/Payment/Review/Dispute/Notification; `owner_id TEXT` вместо `users`. Это копит долг и блокирует аналитику/выплаты.

### 🟡 P2 — Полнота приложений
10. **telegram-bot пуст** — нет точки входа mini-app, initData-auth, уведомлений (статус заказа, новое сообщение, готово заключение), inline «Принять/Отклонить» для врача.
11. **couples — не mini-app.** Нет `@twa-dev/sdk` (`WebApp.ready/expand/colorScheme`, BackButton, тема Telegram). Сейчас это обычное web-SPA.
12. **vendor/admin: экраны без бэкенда.** Services/Schedule/Finance/Reviews (vendor) и Disputes/Finance/часть модерации (admin) — UI на моках; нет таблиц `vendor_services/slots/payouts`, `reviews`, `disputes`, `payments`.
13. **`packages/shared` не выполняет роль.** По `CLAUDE.md` это «общие типы, утилиты, локализация, бренд», по факту — только иконки. Статус-лейблы/цвета, типы `Consultation/Order`, i18n-ключи дублируются → дрейф (напр. цвета статусов уже заданы по-разному в трёх местах).

### 🟢 P3 — Качество и эксплуатация
14. **Тестов нет вообще** (`npm test` → no-op). Нет CI.
15. **Языковой паритет неполон.** couples двуязычен; vendor/admin — преимущественно хардкод RU. Спека/DESIGN требуют uz везде.
16. **a11y не подтверждена.** DESIGN требует AA-контраст, видимый фокус, `prefers-reduced-motion`, цель ≥44px — нужно проверить и починить, особенно во `video.html` и кастомных контролах.
17. **Эксплуатация.** Нет валидации env (падение без `DATABASE_URL`), нет мониторинга ошибок/структурного лога. WS-сигналинг и лобби входящих звонков — **in-memory**, не масштабируются за пределы одного процесса.
18. **Хрупкость воркспейса.** Симлинк `node_modules/@ht/shared` ломается на синхро-диске (см. заметку в `CLAUDE.md`) → сборки падали. Нужна устойчивая линковка/скрипт восстановления.

## 4. Приоритеты и порядок работ

Делать **строго по приоритету**, тонкими сквозными срезами, не ломая существующие контракты и дизайн.

```
P0  Безопасность  → хеш паролей, initData-auth, проверка владения, токен-сессия admin, webhook-подпись
P1  Платежи+заказ → единый lifecycle заказа, реальный провайдер (1 шт. для начала), эскроу/возвраты
P2  Полнота       → telegram-bot, couples→mini-app, бэкенд под vendor/admin-моки, консолидация shared
P3  Качество      → тесты+CI, uz-паритет, a11y-аудит, env-валидация/логи, устойчивость воркспейса
```

## 5. Milestones (с задачами и критериями приёмки)

> Каждый Milestone — рабочий PR: `docker compose up -d` поднимается, существующие потоки не сломаны, экраны на uz/ru, состояния loading/empty/error на месте.

### M1 (P0) — Аутентификация и хранение секретов
1. **Хеширование паролей:** ввести `bcrypt`. Миграция: при сидинге писать хеши; логин (`vendor`, `admin`) сравнивает через `bcrypt.compare`. Plaintext-сравнения убрать.
2. **Telegram initData:** `POST /api/auth/telegram { initData }` → валидация HMAC (`TELEGRAM_BOT_TOKEN`), апсерт в новую таблицу `users (id, telegram_id, phone, name, locale, roles[], status)`, привязка `owner_id ↔ user_id`, выдача короткоживущего токена (JWT/подписанный).
3. **Сессии/токены вместо заголовков-паролей:** admin и vendor получают токен при логине; middleware проверяет токен, не пароль на каждом запросе.
4. **Проверка владения (IDOR):** middleware, который сверяет владельца ресурса (`pets`, `consultations`, `orders`) с субъектом токена; чужое → 403.
5. **Гигиена:** убрать `_dev_code` из прод-ответов (оставить только при `NODE_ENV=development`); валидация подписи в `/payments/webhook/:provider`.

**Приёмка:** ни одного plaintext-пароля в коде/БД; поддельный `initData` отклоняется; запрос к чужому `pet/order` → 403; admin/vendor работают по токену; демо-аккаунты по-прежнему логинятся.

### M2 (P1) — Единый жизненный цикл заказа + один реальный провайдер
1. **`orders` как источник истины** для статуса: эндпоинты переходов `accepted/in_progress/completed/cancelled/refunded` с валидацией допустимых переходов (см. `service-spec.md §8.1`); синхронизация со `consultations.status`.
2. **Таблица `payments`** (order_id, provider, amount, status, external_ref, paid_at) и связь с заказом.
3. **Реальная интеграция — начать с одного провайдера** (рекомендуется Payme или Click): `POST /api/payments { order_id, provider } → { checkout_url }`, обработка webhook с проверкой подписи → перевод `payment` и `order` в `paid`.
4. **Эскроу-логика (MVP):** средства «held» при `paid`, расчёт выплаты подрядчику после `completed` за вычетом `commission_rate` (дефолт 15%); возвраты при `rejected/cancelled`.
5. Демо-`/simulate` оставить **только для dev**, спрятать за `NODE_ENV`.

**Приёмка:** заказ проходит `created→paid→accepted→in_progress→completed`; недопустимый переход → 400; реальный sandbox-платёж одного провайдера доходит до `paid` через webhook; возврат переводит в `refunded`.

### M3 (P2) — telegram-bot и couples как mini-app
1. **`apps/telegram-bot`:** `/start` (язык + кнопка «Открыть приложение»), привязка телефона, уведомления (статус заказа, новое сообщение, готово заключение), для врача — inline «Принять/Отклонить» (callback → переход статуса).
2. **couples → mini-app:** подключить `@twa-dev/sdk` (`ready/expand`, тема из `colorScheme`, нативный `BackButton`), авторизация через `POST /api/auth/telegram` из M1, оплата внутри mini-app (checkout из M2).
3. Сохранить уже готовый uz/ru и существующие экраны; не переписывать то, что работает.

**Приёмка:** бот отвечает на `/start` на uz/ru; mini-app открывается из бота с валидной auth; врач принимает заказ из Telegram, статус меняется; видеозвонок/чат продолжают работать.

### M4 (P2) — Бэкенд под существующие моки vendor/admin + консолидация shared
1. **Таблицы и роуты** по брифам: `vendor_services`, `vendor_slots`, `vendor_payouts`, `reviews`, `disputes` (+ роуты из `vendor.md §6` и `admin.md §7`). Подключить к уже существующим экранам (убрать моки).
2. **`packages/shared` как единый источник:** вынести туда типы (`Consultation/Order/Review/...`), статус-лейблы и **цвета статусов**, i18n-ядро (uz/ru ключи общего назначения), бренд-токены. Приложения импортируют из shared, дубли удалить.
3. **Audit-лог на КАЖДУЮ мутацию admin** (сейчас 3 места) — довести покрытие.

**Приёмка:** экраны vendor (Services/Schedule/Finance/Reviews) и admin (Disputes/Finance/Reviews) работают на реальных данных; статус-чипы берутся из shared и одинаковы во всех трёх app; каждая admin-мутация пишет в `audit_log`.

### M5 (P3) — Качество, языковой паритет, эксплуатация
1. **Тесты:** настроить тест-раннер (Vitest/Jest + supertest). Покрыть: auth (initData/хеши/токены), переходы статусов заказа, проверка владения, расчёт выплат, webhook-подпись. Добавить CI (GitHub Actions) с `lint + test + build`.
2. **uz-паритет** для vendor/admin: вынести строки в i18n, добавить uz; убрать хардкод.
3. **a11y-аудит:** контраст пар токенов (AA), видимый фокус, `prefers-reduced-motion` (особенно `video.html` и кастомные кнопки), цели ≥44px. Починить найденное.
4. **Эксплуатация:** валидация env при старте (fail-fast с понятной ошибкой), структурный лог + перехват ошибок, заметка/скрипт по восстановлению воркспейс-симлинка (`@ht/shared`). Опционально: вынести WS-сигналинг/лобби из памяти (Redis pub/sub) для масштабирования.

**Приёмка:** `npm test` зелёный в CI; vendor/admin переключают uz/ru без хардкода; a11y-чек пройден по чек-листу; API не стартует без обязательных env с внятной ошибкой.
