# Промт-проект: vendor — сторона предложения (HappyTails)

> («Реализуй по docs/prompts/vendor.md, начни с Milestone 1»). Сначала прочитай `CLAUDE.md`,
> `PRODUCT.md`, `DESIGN.md`, `docs/service-spec.md` (§3 данные, §5 vendor, §8 процессы).

## Роль и цель

Ты строишь **кабинет подрядчика vendor** — web-приложение, где ветеринары, кинологи, зоомагазины и эксперты принимают заказы, оказывают услуги и получают выплаты. Цель: дать подрядчику поток заказов и прозрачные финансы, а платформе — контроль качества через верификацию. Фокус волны 1 — **ветеринары** (онлайн-консультации).

## Контекст

- Стратегия и принципы — `PRODUCT.md`. Доверие строится верификацией и качеством подрядчиков.
- Визуал, токены, компоненты, бренд — `DESIGN.md` (тёплый коралл/оранж, Onest, крупные скругления). vendor — это product-UI: плотнее couples, упор на ясность и формы.
- Локали uz/ru, статусы заказа, эскроу-логика выплат — канон из `service-spec.md`.

## Технологии

- React web (desktop-first, контейнер ~1200px; адаптив до планшета).
- Данные — через API `services/api` с ролевым доступом (роль `vendor`, `vendor_staff`).
- Уведомления о заказах — в реальном времени через Telegram (`apps/telegram-bot`) с inline-кнопками «Принять/Отклонить».
- Токены/типы/i18n — из `packages/shared`.

## Объём MVP (волна 1)

Регистрация и верификация · профиль подрядчика · CRUD услуг (uz/ru) · расписание/доступность · входящие/активные/завершённые заказы (принять/отклонить/завершить) · чат консультаций · финансы (баланс, комиссия, история выплат, реквизиты) · отзывы (просмотр/ответ). Полный список — `service-spec.md` §5.4.

## Экраны (web-кабинет)

1. Регистрация и верификация — документы/лицензия, статус заявки (pending/verified/rejected).
2. Дашборд — заказы сегодня, доход, рейтинг, отзывы.
3. Услуги — CRUD, локализация uz/ru, цена, длительность, активность.
4. Расписание/доступность — слоты для онлайн-консультаций.
5. Заказы — входящие/в работе/завершённые; принять/отклонить/завершить; статусы.
6. Чат консультаций — текст/фото/файлы, итоговое заключение.
7. Финансы — баланс, ставка комиссии, история выплат, реквизиты, запрос выплаты.
8. Отзывы — просмотр и ответ.
9. Профиль — описание, город, тип, документы.

## Данные (что нужно vendor; полная модель — service-spec.md §3)

`User` (роли vendor/vendor_staff), `Vendor` (type, verification_status, rating, commission_rate, payout_details), `Service`, `Order`, `Consultation`, `Payment`, `Review`.

## API-контракты (минимум для среза)

- `POST /api/vendors/register` · `POST /api/vendors/:id/verification` (загрузка документов) · статус.
- `GET/POST/PUT /api/vendors/:id/services` — управление услугами (с локализацией).
- `GET/PUT /api/vendors/:id/availability` — расписание.
- `GET /api/vendors/:id/orders?status=` — заказы подрядчика.
- `POST /api/orders/:id/accept` · `/reject` · `/complete` — смена статуса (правила из §8.1).
- `GET /api/consultations/:id/messages` · `POST .../messages` — чат.
- `GET /api/vendors/:id/balance` · `POST /api/vendors/:id/payouts` — финансы/выплаты.
- `GET /api/vendors/:id/reviews` · `POST /api/reviews/:id/reply`.

## Бизнес-правила

- Верификация обязательна до публикации услуг (нельзя активировать `Service` при `verification_status != verified`).
- Комиссия на уровне подрядчика/категории (`commission_rate`); выплата после `completed` за вычетом комиссии (эскроу — §8.2).
- Переходы статусов только разрешённые: `paid → accepted → in_progress → completed`; `accepted/paid → rejected → refunded`.

## Дизайн и качество

- Токены и компоненты из `DESIGN.md`; формы крупные, label явные, ошибки текст+иконка (не только цвет).
- Строки uz/ru через i18n; запас +30% длины. a11y AA, видимый фокус, `prefers-reduced-motion`.
- Состояния loading/empty/error в каждом разделе; пустой дашборд — с подсказкой «создайте первую услугу».

## Definition of Done (для среза)

Подрядчик проходит путь: регистрация → верификация (мок-аппрув) → создание услуги → получение заказа (уведомление в Telegram) → принятие → консультация → завершение → отражение в финансах. uz/ru, корректные статусы, проверенный контраст.

## Milestone 1 — приём и выполнение заказа

1. Каркас web-кабинета + вход с ролью `vendor`.
2. Регистрация + экран верификации (статусы; аппрув можно из admin/мок).
3. CRUD одной услуги «онлайн-консультация» (uz/ru, цена).
4. Список входящих заказов + принять/отклонить; уведомление в Telegram (inline).
5. Чат консультации + завершение (`completed`) с заключением.
6. Простой экран финансов: баланс/комиссия по завершённому заказу. Состояния loading/empty/error.

## Не-цели (волна 2)

Кинолог/дрессировка, страхование как продукт подрядчика, расширенные выплаты и аналитика, мультифилиальность. Не реализовывать сейчас.
