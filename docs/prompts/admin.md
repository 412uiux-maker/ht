# Промт-проект: admin — платформа / администрирование (HappyTails)

> Как использовать: открой Claude Code в корне репозитория и дай ему этот файл как задание
> («Реализуй по docs/prompts/admin.md, начни с Milestone 1»). Сначала прочитай `CLAUDE.md`,
> `PRODUCT.md`, `DESIGN.md`, `docs/service-spec.md` (§2 роли, §3 данные, §6 admin, §8 процессы).

## Роль и цель

Ты строишь **внутреннюю панель admin** — web-инструмент оператора платформы. Цель: операционное управление качеством и деньгами — верификация подрядчиков, модерация, разбор споров, выплаты, контент и аналитика. admin делает возможной работу couples и vendor (верификация, возвраты, контент).

## Контекст

- Принципы и приоритеты — `PRODUCT.md`. admin косвенно поддерживает доверие (быстрая верификация, честные возвраты).
- Визуал — `DESIGN.md`, но admin — плотный data-UI: таблицы, фильтры, очереди; те же токены/шрифт, меньше «маркетингового» декора, упор на читаемость и скорость.
- Роли/права (RBAC), статусы заказа, эскроу/возвраты — канон из `service-spec.md`.

## Технологии

- React web (desktop, контейнер ~1200px+, таблицы с фильтрами/пагинацией).
- Данные — через API `services/api` с ролевым доступом (`moderator`, `support`, `admin`).
- Гранулярные права (RBAC) и **аудит-лог** всех действий администраторов.
- Токены/типы/i18n — из `packages/shared` (интерфейс admin допустимо ru-first, но строки через i18n).

## Объём MVP (волна 1)

Верификация подрядчиков · модерация (услуги, отзывы, контент) · заказы (поиск, просмотр, отмена/возврат) · споры/поддержка · финансы (транзакции, комиссии, выплаты, сверка) · контент (CRUD гайдов/курсов/мест, публикация, локализация) · добрые дела · пользователи и роли (RBAC) · аналитика · настройки. Полный список — `service-spec.md` §6.2.

## Экраны/модули

1. Очередь верификации — заявки подрядчиков, проверка документов, approve/reject с комментарием.
2. Модерация — услуги, отзывы, пользовательский контент.
3. Заказы — поиск/фильтр, детали, ручная отмена/возврат.
4. Споры/поддержка — тикеты, переписка, решение по возвратам.
5. Финансы — транзакции, комиссии, выплаты подрядчикам, сверка с провайдерами.
6. Контент — CRUD статей/курсов/гайдов/мест, публикация, локализация uz/ru.
7. Добрые дела — инициативы, сбор/распределение.
8. Пользователи и роли — управление RBAC.
9. Аналитика — GMV, заказы, средний чек, доля платящих, удержание, CAC-данные, конверсия подбора корма, рейтинг подрядчиков, LTV/CAC.
10. Настройки — комиссии, категории, провайдеры оплаты, шаблоны уведомлений.

## Данные (что нужно admin; полная модель — service-spec.md §3)

`User` (все роли), `Vendor`, `Service`, `Order`, `Payment`, `Review`, `ContentItem`, `Notification`, `AuditLog` (новое: actor, action, target, timestamp).

## API-контракты (минимум для среза)

- `GET /api/admin/vendors?verification_status=pending` · `POST /api/admin/vendors/:id/verify` (approve/reject + comment).
- `GET /api/admin/moderation?type=service|review|content` · `POST .../moderate`.
- `GET /api/admin/orders?status=&q=` · `POST /api/admin/orders/:id/refund` (возврат → §8.2).
- `GET /api/admin/disputes` · `POST /api/admin/disputes/:id/resolve`.
- `GET /api/admin/finance/transactions` · `GET /api/admin/finance/payouts`.
- `GET/POST/PUT /api/admin/content` — контент с локализацией.
- `GET/POST /api/admin/users` · `PUT /api/admin/users/:id/roles` — RBAC.
- `GET /api/admin/analytics?metric=&from=&to=`.
- Все мутации пишут `AuditLog`.

## Бизнес-правила

- RBAC: `moderator` — модерация/верификация; `support` — споры/возвраты; `admin` — всё + финансы/настройки. Действия вне роли — 403.
- Возврат меняет `Payment.refunded` и `Order` по правилам §8 (`cancelled/rejected → refunded`).
- Каждое действие администратора фиксируется в аудит-логе (кто, что, над чем, когда).

## Дизайн и качество

- Токены/шрифт из `DESIGN.md`; плотные таблицы с сортировкой/фильтрами/пагинацией; статус-чипы единые с остальными приложениями.
- a11y AA, видимый фокус, клавиатурная навигация по таблицам. Состояния loading/empty/error в каждом модуле.

## Definition of Done (для среза)

Оператор с ролью входит, видит очередь верификации, аппрувит/реджектит подрядчика (что разблокирует публикацию услуг в vendor), находит заказ и оформляет возврат; все действия в аудит-логе; права по ролям соблюдаются.

## Milestone 1 — верификация + возвраты

1. Каркас admin + вход с ролью и базовый RBAC (3 роли).
2. Очередь верификации подрядчиков + approve/reject с комментарием (разблокирует vendor).
3. Список заказов с поиском/фильтром по статусу + ручной возврат (`refunded`).
4. Аудит-лог действий (запись + простой просмотр).
5. Состояния loading/empty/error; проверка прав (403 вне роли).

## Не-цели (волна 2)

Полная аналитика-дашборд с графиками, расширенная сверка платежей, управление добрыми делами на потоке, A/B настроек. Не реализовывать сейчас.
