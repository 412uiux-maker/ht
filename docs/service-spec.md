# Техническая спецификация платформы для питомцев (Узбекистан)

**Версия:** 0.1 (черновик для оценки разработки)
**Дата:** 2026-06-16
**Назначение:** описание сервиса и его четырёх приложений для команды разработки и подрядчиков.

---

## 1. Обзор продукта

Платформа — маркетплейс услуг и контента о здоровье, питании и уходе за питомцами. Соединяет **владельцев питомцев** (сторона спроса) с **подрядчиками** (ветеринары, кинологи, зоомагазины, эксперты — сторона предложения) и предоставляет обучающий контент.

### Четыре приложения (стороны системы)

| # | Приложение | Роль | Платформа |
|---|------------|------|-----------|
| 1 | **Couples** (клиент) | Сторона спроса — владельцы питомцев | Mini-app в Telegram + (позже) web/mobile |
| 2 | **Vendor** | Сторона предложения — подрядчики | Web-кабинет + Telegram-уведомления |
| 3 | **Admin** | Внутренняя панель оператора | Web |
| 4 | **Telegram** | Бот + точка входа в mini-app | Telegram Bot API |

> Термин «couples» используется как внутреннее название клиентского приложения = сторона спроса (владельцы питомцев). Везде ниже трактуется именно так.

### Архитектурный принцип

Единый backend (API) обслуживает все четыре фронтенда. Telegram — основной канал входа на старте; couples mini-app работает поверх него. Vendor и Admin — отдельные web-кабинеты, использующие тот же API с ролевым доступом.

```
                 ┌──────────────┐
   Telegram ────▶│              │
   couples  ────▶│   API / BFF  │◀──── Vendor (web)
   (mini-app)    │   + Auth     │◀──── Admin (web)
                 └──────┬───────┘
                        │
          ┌─────────────┼──────────────┐
        DB (PostgreSQL)  Payments     Notifications
                        (Click/Payme/  (Telegram, push,
                         Uzum Pay)      SMS)
```

## 2. Роли и права доступа

| Роль | Описание | Доступ |
|------|----------|--------|
| `client` | Владелец питомца | couples mini-app |
| `vendor` | Верифицированный подрядчик | vendor-кабинет |
| `vendor_staff` | Сотрудник подрядчика (напр. ветеринар в клинике) | vendor-кабинет, ограниченно |
| `moderator` | Модерация контента и заявок | admin |
| `support` | Поддержка, споры | admin |
| `admin` | Полный доступ, настройки, финансы | admin |

Авторизация: единый `user_id`, роли через RBAC. Один человек может иметь несколько ролей (напр. ветеринар, который сам владелец питомца).

## 3. Модель данных (ключевые сущности)

Перечень основных сущностей и связей. Поля — минимально необходимый набор для MVP.

### User
`id, phone, telegram_id, name, locale (uz|ru), roles[], created_at, status`

### Pet (питомец)
`id, owner_id → User, species (dog|cat|other), breed, name, sex, birth_date, weight, notes, avatar`

### Vendor (подрядчик)
`id, user_id → User, type (vet|cynologist|shop|expert), display_name, description, city, verification_status (pending|verified|rejected), rating, commission_rate, payout_details`

### Service (услуга/предложение)
`id, vendor_id → Vendor, category (vet_online|food_pick|training|...), title, description, price, duration_min, format (online|offline), is_active, locale_content{uz,ru}`

### Order (заказ)
`id, client_id → User, pet_id → Pet, service_id → Service, vendor_id → Vendor, status, scheduled_at, price, commission, payout_amount, payment_id, created_at`

### Consultation (для онлайн-ветеринара)
`id, order_id → Order, chat_thread_id, started_at, ended_at, summary, attachments[]`

### Payment
`id, order_id → Order, provider (click|payme|uzum), amount, status (pending|paid|refunded|failed), external_ref, paid_at`

### Review (отзыв)
`id, order_id → Order, client_id, vendor_id, rating (1–5), text, created_at, moderation_status`

### ContentItem (обучающий контент)
`id, type (article|course|guide|place), category, title, body, locale{uz,ru}, media[], status (draft|published), author_id`

### Notification
`id, user_id, channel (telegram|push|sms), template, payload, status, sent_at`

## 4. Couples — сторона спроса (клиент)

### 4.1 Цель
Дать владельцу питомца быстро решить боль: получить консультацию, подобрать корм, научиться ухаживать, найти проверенного подрядчика — с понятной оплатой и гарантией.

### 4.2 Основные сценарии (user stories)

- Как владелец, я хочу **создать профиль питомца** (вид, порода, возраст, вес), чтобы рекомендации были персональными.
- Как владелец, я хочу **записаться на онлайн-консультацию ветеринара** и оплатить её через Click/Payme, чтобы получить помощь не выходя из дома.
- Как владелец, я хочу **переписываться/созваниваться с ветеринаром** в рамках заказа и получить итоговое заключение.
- Как владелец, я хочу **подобрать корм** по анкете питомца и перейти к покупке у партнёра.
- Как владелец, я хочу **проходить курс/школу владельца** и получать чек-листы (подготовка к ветврачу, экстренная помощь).
- Как владелец, я хочу **видеть отзывы и рейтинг подрядчика** перед покупкой, чтобы доверять выбору.
- Как владелец, я хочу **оставить отзыв** после услуги.
- Как владелец, я хочу **участвовать в «добрых делах»** (помочь животным), чтобы вовлекаться в сообщество.
- Как владелец, я хочу **находить места, где можно с питомцем** (кафе, парки), на карте.

### 4.3 Экраны (mini-app)

1. **Онбординг** — выбор языка (uz/ru), вход по Telegram, создание профиля питомца.
2. **Главная** — персональная лента: рекомендованные услуги, контент, баннер «добрых дел».
3. **Каталог услуг** — фильтры по категории, городу, цене, рейтингу.
4. **Карточка услуги/подрядчика** — описание, цена, отзывы, кнопка заказа.
5. **Оформление заказа** — выбор питомца, времени, оплата.
6. **Чат консультации** — текст, фото, файлы, статус, заключение.
7. **Мои заказы** — активные/история, статусы, повтор заказа.
8. **Подбор корма** — анкета → результат → переход к покупке.
9. **Обучение** — курсы, гайды, чек-листы; прогресс.
10. **Добрые дела** — список инициатив, участие/донат.
11. **Места с питомцем** — карта/список.
12. **Профиль** — питомцы, язык, способы оплаты, история.

### 4.4 Ключевые требования
- Полная локализация **uz/ru** (контент и интерфейс).
- Оплата: Click, Payme, Uzum Pay.
- Доверие: верифицированные подрядчики (бейдж), отзывы, понятная политика возврата/гарантии.

## 5. Vendor — сторона предложения (подрядчики)

### 5.1 Цель
Дать подрядчику инструменты для приёма заказов, оказания услуги и получения выплат, а платформе — контроль качества.

### 5.2 Типы подрядчиков
Ветеринар (онлайн-консультации), кинолог, зоомагазин/бренд корма (подбор, продажа), эксперт-автор контента.

### 5.3 User stories
- Как подрядчик, я хочу **пройти регистрацию и верификацию** (документы, лицензия), чтобы получить бейдж доверия.
- Как подрядчик, я хочу **создавать и редактировать услуги** с ценой и описанием на uz/ru.
- Как подрядчик, я хочу **управлять расписанием/доступностью** для онлайн-консультаций.
- Как подрядчик, я хочу **получать уведомление о новом заказе** в Telegram и принимать/отклонять его.
- Как ветеринар, я хочу **вести консультацию в чате** и оставить заключение.
- Как подрядчик, я хочу **видеть статистику** (заказы, доход, рейтинг, отзывы).
- Как подрядчик, я хочу **запрашивать выплату** и видеть баланс/комиссию.

### 5.4 Экраны (web-кабинет)
1. Регистрация и верификация (статус заявки).
2. Дашборд — заказы сегодня, доход, рейтинг.
3. Услуги — CRUD, локализация, активность.
4. Расписание/доступность.
5. Заказы — входящие, в работе, завершённые; принять/отклонить/завершить.
6. Чат консультаций.
7. Финансы — баланс, комиссия, история выплат, реквизиты.
8. Отзывы — просмотр, ответ.
9. Профиль подрядчика — описание, город, документы.

### 5.5 Требования
- Верификация обязательна до публикации услуг.
- Комиссия платформы настраивается на уровне подрядчика/категории (`commission_rate`).
- Уведомления о заказах — в реальном времени через Telegram.

## 6. Admin — внутренняя панель

### 6.1 Цель
Операционное управление: модерация, поддержка, финансы, аналитика, контент.

### 6.2 Модули
1. **Верификация подрядчиков** — очередь заявок, проверка документов, approve/reject с комментарием.
2. **Модерация** — услуги, отзывы, пользовательский контент.
3. **Заказы** — поиск, просмотр, ручное вмешательство, отмена/возврат.
4. **Споры/поддержка** — тикеты, переписка, решение по возвратам.
5. **Финансы** — транзакции, комиссии, выплаты подрядчикам, сверка с провайдерами.
6. **Контент** — CRUD статей/курсов/гайдов/мест, публикация, локализация.
7. **Добрые дела** — управление инициативами, сбор/распределение.
8. **Пользователи и роли** — управление RBAC.
9. **Аналитика** — ключевые метрики (см. §8): заказы, GMV, CAC-данные, удержание, конверсия подбора корма.
10. **Настройки** — комиссии, категории, провайдеры оплаты, шаблоны уведомлений.

### 6.3 Требования
- Полный аудит-лог действий администраторов.
- Гранулярные права по ролям (moderator/support/admin).

## 7. Telegram — бот + mini-app

### 7.1 Роль
Основная точка входа на старте. Бот = идентификация, уведомления, навигация; mini-app = полноценный клиентский интерфейс (couples).

### 7.2 Функции бота
- `/start` — приветствие, выбор языка, кнопка «Открыть приложение» (mini-app).
- Авторизация через Telegram (telegram_id → User), привязка телефона.
- **Уведомления:** статус заказа, напоминание о консультации, новое сообщение в чате, готово заключение, выплата (для vendor).
- Быстрые действия: «Мои заказы», «Задать вопрос ветеринару», «Добрые дела».
- Для подрядчиков: уведомления о новых заказах с inline-кнопками «Принять/Отклонить».

### 7.3 Mini-app
- Запускается из бота, рендерит интерфейс couples (§4).
- Авторизация через Telegram initData (валидация на backend).
- Оплата внутри mini-app через провайдеров (Click/Payme/Uzum) — web-checkout/redirect.
- Темизация под Telegram (light/dark), адаптив под мобильные.

### 7.4 Требования
- Валидация подписи `initData` на сервере (безопасность).
- Бесшовный переход бот ↔ mini-app.
- Поддержка uz/ru на уровне сообщений бота.

## 8. Сквозные процессы

### 8.1 Жизненный цикл заказа (статусы)

```
created → paid → accepted → in_progress → completed → reviewed
                    │
                    ├─▶ rejected (подрядчик отклонил) → refunded
                    └─▶ cancelled (клиент/админ) → refunded
```

| Статус | Описание | Кто меняет |
|--------|----------|------------|
| `created` | Заказ оформлен, ждёт оплаты | client |
| `paid` | Оплата прошла, ждёт подрядчика | система (payment webhook) |
| `accepted` | Подрядчик принял | vendor |
| `rejected` | Подрядчик отклонил → возврат | vendor |
| `in_progress` | Услуга оказывается (консультация идёт) | vendor |
| `completed` | Услуга оказана | vendor |
| `cancelled` | Отмена до выполнения → возврат | client/admin |
| `reviewed` | Клиент оставил отзыв | client |
| `refunded` | Возврат средств выполнен | система/admin |

### 8.2 Платежи
- Провайдеры: **Click, Payme, Uzum Pay** (через их API/checkout).
- Эскроу-логика: средства фиксируются при `paid`, выплата подрядчику после `completed` за вычетом комиссии.
- Webhook от провайдера → смена статуса `Payment` и `Order`.
- Возвраты при `rejected/cancelled`.

### 8.3 Уведомления
Единый сервис уведомлений, каналы: Telegram (основной), push (для будущего приложения), SMS (критичное, опционально). Шаблоны с локализацией uz/ru.

### 8.4 Ключевые метрики (для Admin/аналитики)
GMV, число заказов, средний чек, доля платящих, удержание (retention), CAC-данные по каналам, конверсия «подбор корма → покупка», рейтинг подрядчиков, LTV/CAC.

## 9. Нефункциональные требования

- **Локализация:** uz/ru на всех уровнях (UI, контент, уведомления). Архитектура контента — мультиязычные поля.
- **Стек (предложение):** Backend — Node.js/NestJS или Python/FastAPI; БД — PostgreSQL; кэш/очереди — Redis; mini-app — React + Telegram Web Apps SDK; vendor/admin — React. Уточняется командой.
- **Безопасность:** валидация Telegram initData, RBAC, аудит-лог админа, шифрование платёжных реквизитов, GDPR-подобная обработка перс. данных, соответствие требованиям РУз по данным.
- **Производительность:** mini-app должен открываться быстро на среднем мобильном; оптимизация под 3G/4G.
- **Масштабирование:** старт с Ташкента, готовность к регионам (города как измерение в каталоге).
- **Наблюдаемость:** логирование, мониторинг ошибок, дашборд статусов платежей.

## 10. Объём MVP (приоритеты)

**Волна 1 (MVP):**
- couples mini-app: профиль питомца, каталог, заказ + оплата, чат онлайн-консультации, обучение (гайды/чек-листы), подбор корма, добрые дела, места с питомцем.
- vendor: регистрация/верификация, услуги, заказы, чат, базовые финансы — фокус на **ветеринарах**.
- admin: верификация, заказы, модерация, финансы, контент, аналитика.
- telegram: бот (вход, уведомления, vendor inline-действия) + mini-app.
- Платежи: Click + Payme + Uzum Pay.

**Волна 2 (после пилота):**
- Сравнение/проверка кормов, кинолог онлайн + видеоуроки, страхование, диагностика по симптомам (с осторожностью — мед. ответственность), AR-игра, расширенная аналитика и выплаты.
